# 公众号采集 · 账号池（Account Pool）功能设计文档

> 面向实现 Agent 的设计与施工说明。阅读对象：负责落地此功能的工程 Agent。
> 目标：为「微信公众号采集」增加**多账号池**能力——采集过程中自动轮换账号、被风控的账号自动踢出账号池并在页面提示、登录页改造为「账号池」管理页。

---

## 1. 背景与现状

### 1.1 当前单账号架构

目前整个微信公众号采集只支持**单个账号**，凭证保存在单个 JSON 文件：

- 文件：`data/wechat_mp_config.json`
- 结构：
  ```json
  {
    "token": "xxxxxxxx",
    "cookie_str": "slave_sid=...; data_bizuin=...; ...",
    "cookies": [ { "name": "...", "value": "...", "domain": "...", ... } ],
    "save_time": 1718000000.0,
    "account_info": { "nickname": "某公众号", "avatar": "https://..." }
  }
  ```
- 常量定义：`backend/config.py` → `CONFIG_FILE = DATA_DIR / "wechat_mp_config.json"`

### 1.2 凭证的消费点（必须全部改造）

所有采集请求都从同一个文件读取 `token` + `cookie_str`，再拼到 `Cookie` 头里发请求。当前消费点：

| 文件 | 函数 | 说明 |
|------|------|------|
| `backend/accounts.py` | `_get_session()`（L18） | 搜索公众号 `searchbiz` |
| `backend/articles.py` | `_get_session()`（L28）、`_fetch_articles_page()`（L36） | 文章列表 `appmsgpublish`、`appmsg` |
| `backend/auth.py` | `get_status()`、`check_credentials()` | 登录状态/凭证校验 |
| `backend/rss_scheduler.py` | `_fetch_for_account()` | 后台 RSS 定时抓取（间接调用 `articles._fetch_articles_page`） |

> ⚠️ `accounts.py` 与 `articles.py` 各有一份**重复**的 `_get_session()`。本次改造要统一收口到账号池模块。

### 1.3 微信风控/失效返回码（关键）

采集请求返回 JSON，`base_resp.ret` 字段含义（已从 `we-mp-rss/core/wx/` 与本项目代码确认）：

| ret | 含义 | 处理策略 |
|-----|------|----------|
| `0` | 成功 | 标记账号一次成功，清零失败计数 |
| `200013` | **频率控制（风控）** | 账号进入冷却；累计触发达阈值则**踢出账号池** |
| `200003` | **登录态失效 / session 过期** | 账号标记为 `invalid`，**踢出账号池**，需重新扫码 |
| 其他非 0 | 一般业务错误 | 记一次失败，不一定踢出 |
| HTTP != 200 | 网络/代理问题 | 走代理失败逻辑，不计入账号风控（除非确认是账号问题） |

### 1.4 已有的「池」范式可直接复用（重要）

项目里**已经有一套成熟的"池 + 轮换 + 失败计数 + 冷却"实现**——代理池，位于 `backend/config.py`：

- `get_proxy_url()`（L157）：轮换选择、按 `(failures, last_used)` 排序挑最优节点、全部冷却时选最快恢复的
- `report_proxy_status()`（L270）：成功清零、失败累加、连续失败 5 次进入 600s 冷却
- 全局状态：`_proxy_states` + `_proxy_lock`

**账号池请直接照搬这套范式**（数据结构、调度算法、线程锁、状态上报回调），保持代码风格一致、降低评审成本。

## 2. 需求拆解

用户原始需求：

1. **账号池**：采集支持多账号，采集过程中**自动切换账号**，避免单账号被风控。
2. **自动踢出**：账号被风控（`200013` 累计 / `200003`）时，自动从可用池中踢出。
3. **页面提示**：账号被踢出时在页面给出明确提示（哪个账号、什么原因）。
4. **登录页改造**：把现有"扫码登录"页改造成"**账号池**"管理页——可添加多个账号、查看每个账号状态、手动移除/重新登录。

## 3. 总体设计

### 3.1 设计原则

- **复用优先**：账号池调度 = 代理池调度的同构实现。
- **最小侵入**：所有采集请求改为「从账号池借一个账号」，而不是各自读 `CONFIG_FILE`。
- **向后兼容**：首次启动时，若检测到旧的 `wechat_mp_config.json`，自动迁移为账号池里的第一个账号（见 §7）。
- **线程安全**：池状态被 Flask 多线程 + RSS 后台线程并发访问，必须加锁。

### 3.2 数据流（采集一次请求）

```
采集请求 (articles / accounts / rss_scheduler)
   │
   ├─ pool.acquire()  ──►  选出一个 status=active 且不在冷却中的账号
   │                        （按 失败次数升序 + 最久未用 排序）
   │
   ├─ 用该账号 token+cookie 发起微信 API 请求
   │
   └─ 根据 base_resp.ret 调用 pool.report(account_id, ret, http_ok)
            ├─ ret==0        → 成功，清零失败计数
            ├─ ret==200013   → 风控：失败+1，进入冷却；累计达阈值→踢出
            ├─ ret==200003   → 失效：直接踢出（status=invalid）
            └─ 其他          → 失败+1
```

### 3.3 账号"踢出"的语义

"踢出账号池"= 把账号 `status` 从 `active` 改为非可用态，使 `acquire()` 不再返回它，但**记录仍保留**在池里（前端能看到它被踢出 + 原因），用户可手动「重新登录」复活或「删除」。

| status | 含义 | 是否参与采集 | 来源 |
|--------|------|--------------|------|
| `active` | 正常可用 | ✅ | 登录成功 |
| `cooldown` | 临时冷却（风控触发但未达踢出阈值） | ❌（冷却到期自动恢复 active） | `ret==200013` |
| `banned` | 被风控踢出 | ❌ | `200013` 累计达阈值 |
| `invalid` | 登录态失效被踢出 | ❌ | `ret==200003` / 凭证过期 |

> `cooldown` 是「软踢出」（可自愈），`banned`/`invalid` 是「硬踢出」（需人工处理）。

## 4. 后端设计

### 4.1 新增模块 `backend/account_pool.py`

职责：账号池的存储、调度（acquire）、状态上报（report）、增删改查。**单例**，类比 `rss_scheduler`。

#### 4.1.1 存储文件

- 新增：`data/account_pool.json`
- 在 `backend/config.py` 增加常量：
  ```python
  ACCOUNT_POOL_FILE = DATA_DIR / "account_pool.json"
  ```
- 结构（list[dict]）：
  ```json
  [
    {
      "id": "acc_1718000000_ab12",          // 稳定唯一 id（生成一次，不变）
      "token": "xxxx",
      "cookie_str": "...",
      "cookies": [ ... ],
      "nickname": "某公众号",
      "avatar": "https://...",
      "save_time": 1718000000.0,             // 凭证保存时间（沿用 4 天有效期判断）
      "status": "active",                    // active | cooldown | banned | invalid
      "failures": 0,                          // 连续失败计数（成功清零）
      "risk_hits": 0,                          // 累计风控(200013)次数
      "last_used": 0.0,
      "cooldown_until": 0.0,                  // 冷却结束时间戳
      "last_error": null,                     // 最近一次错误/踢出原因（给前端展示）
      "kicked_time": 0.0                      // 被踢出的时间（banned/invalid 时记录）
    }
  ]
  ```

#### 4.1.2 调度参数（建议常量，放模块顶部）

```python
LOGIN_VALID_SECONDS = 4 * 24 * 60 * 60   # 与 auth.py 保持一致：凭证 4 天有效
COOLDOWN_SECONDS     = 10 * 60           # 单次风控冷却 10 分钟（对齐代理池 600s）
RISK_KICK_THRESHOLD  = 3                 # 累计风控达 3 次 → banned 踢出（可调）
FAILURE_KICK_THRESHOLD = 8               # 连续普通失败达 8 次 → 视为不可用
```

#### 4.1.3 核心接口

```python
class AccountPool:
    def __init__(self): ...
    # ── 存储 ──
    def _load(self) -> list: ...
    def _save(self, accounts: list): ...

    # ── 调度 ──
    def acquire(self) -> dict | None:
        """
        选出一个可用账号并返回其副本（含 token/cookie_str）。
        规则（照搬 get_proxy_url 的逻辑）：
          1. 过滤 status==active 的账号；
          2. 把 cooldown_until 已过期的 cooldown 账号恢复为 active；
          3. 对 active 账号按 (failures, last_used) 升序，取第一个；
          4. 更新其 last_used；
          5. 全部不可用 → 返回 None。
        线程安全（_lock）。
        """

    def report(self, account_id: str, *, ret: int | None = None,
               http_ok: bool = True, error: str | None = None):
        """
        采集结果回写（照搬 report_proxy_status 的范式）：
          - ret==0            → failures=0, status 保持 active
          - ret==200013       → risk_hits+1, failures+1,
                                 status=cooldown, cooldown_until=now+COOLDOWN_SECONDS;
                                 若 risk_hits>=RISK_KICK_THRESHOLD → status=banned（硬踢出）
          - ret==200003       → status=invalid（硬踢出），记录原因
          - http_ok==False    → failures+1（网络层失败，可不立即踢出）
          - 其他非0 ret       → failures+1；failures>=FAILURE_KICK_THRESHOLD → invalid
        每次踢出时写 last_error / kicked_time，供前端提示。
        """

    # ── 增删改查（给 API 层用）──
    def list_accounts(self) -> list: ...            # 返回脱敏列表（不含完整 cookie）
    def add_or_update(self, cred: dict) -> dict:    # 登录成功后写入/更新（按 token 去重）
    def remove(self, account_id: str) -> bool: ...
    def revive(self, account_id: str) -> bool:      # 手动复活：status=active, 清零计数（仅在重新登录后调用）
    def get_active_count(self) -> int: ...
    def pop_kick_events(self) -> list:              # 取出"自上次查询以来新发生的踢出事件"，供前端弹提示
```

#### 4.1.4 踢出事件通知机制（给前端"页面提示"）

为满足「账号被踢出时页面提示」，池内维护一个**踢出事件队列**：

```python
self._kick_events = []   # [{ "id", "nickname", "reason", "time", "status" }]
```

- `report()` 中每次发生 `banned`/`invalid` 时 `append` 一条事件。
- 前端轮询 `GET /api/account-pool/events` 取走并清空（`pop_kick_events`）。
- 前端拿到后用 `Toast.warning(...)` 弹出，例如：
  `「账号【某公众号】触发风控已被移出账号池，请重新登录」`

> 之所以用「事件队列 + 前端轮询」而不是 WebSocket：项目当前全部是 HTTP 轮询风格（登录状态、下载进度 SSE 除外），保持一致最简单。轮询可挂在已有的 `App.checkAuthStatus()`（每 30s）或账号池页自己的定时器上。

### 4.2 统一凭证入口（替换分散的 `_get_session`）

新增一个**全局借用函数**，所有采集点改用它：

```python
# backend/account_pool.py
account_pool = AccountPool()   # 单例

def borrow_session() -> tuple[str, str, str]:
    """
    返回 (account_id, token, cookie_str)。
    无可用账号时抛 RuntimeError("账号池中无可用账号，请先在『账号池』页面登录账号")。
    """
    acc = account_pool.acquire()
    if not acc:
        raise RuntimeError("账号池中无可用账号，请先在『账号池』页面添加/登录账号")
    return acc["id"], acc["token"], acc["cookie_str"]
```

#### 改造点清单（实现 Agent 按此逐个改）

1. **`backend/articles.py`**
   - `_get_session()` → 删除，改调 `borrow_session()`。
   - `_fetch_articles_page()`：
     - 开头 `account_id, token, cookie_str = borrow_session()`；
     - 拿到响应后，解析 `ret` 调 `account_pool.report(account_id, ret=ret, http_ok=...)`；
     - 把现有 `if ret == 200003: raise PermissionError(...)` 之前**先 report**，让池先踢人再抛错。
   - `get_articles_appmsg()` 同理。

2. **`backend/accounts.py`**
   - `_get_session()` → 删除，改调 `borrow_session()`。
   - `search_accounts()`：发请求前 borrow，拿到 `ret` 后 report。

3. **`backend/rss_scheduler.py`**
   - `_fetch_for_account()` 通过 `_fetch_articles_page` 间接走池，无需改；但要确保 `PermissionError`（200003）发生时，本轮订阅标记 `last_error="账号已失效"`（已有逻辑，保留）。
   - 后台抓取若 `acquire()` 返回 None，应跳过本轮并记录，不报错。

4. **`backend/auth.py`**
   - `get_status()` / `check_credentials()`：从「读单文件」改为「读账号池聚合状态」。详见 §4.4。

### 4.3 重试 = 自动切换账号（核心价值）

「采集中自动切换账号」落地在**重试循环**里：单次请求失败/风控时，`report` 把当前账号踢冷却，下次 `acquire` 自然选到另一个账号。建议在 `_fetch_articles_page` 外层封装带切换的重试：

```python
def fetch_with_pool(fetch_once, max_switch=3):
    last_exc = None
    for _ in range(max_switch):
        account_id, token, cookie_str = borrow_session()
        try:
            ret, data = fetch_once(token, cookie_str)   # 返回 (ret, data)
            account_pool.report(account_id, ret=ret)
            if ret == 0:
                return data
            if ret == 200003:        # 失效，换下一个账号重试
                continue
            if ret == 200013:        # 风控，换下一个账号重试
                continue
            # 其他错误：抛出
            raise RuntimeError(f"API错误 (ret={ret})")
        except req.RequestException as e:
            account_pool.report(account_id, http_ok=False, error=str(e))
            last_exc = e
            continue
    raise last_exc or RuntimeError("账号池所有账号均不可用")
```

> 实现 Agent 可按现有 `_fetch_articles_page` 的形态决定是内联还是抽 helper，保持现有函数签名对外不变（`get_articles` 等路由不用改）。

### 4.4 登录与账号池写入

复用现有 `backend/auth.py` 的 Playwright 扫码流程（`_do_login`），**唯一改动**是：登录成功拿到 `token`/`cookies`/`account_info` 后，不再写单文件，而是：

```python
from backend.account_pool import account_pool
account_pool.add_or_update({
    "token": token,
    "cookie_str": cookie_str,
    "cookies": cookies,
    "nickname": nickname or "公众号未命名",
    "avatar": avatar,
    "save_time": time.time(),
})
```

`add_or_update` 按 `token` 或 `nickname` 去重：已存在则更新凭证 + `status=active` + 清零计数（即"重新登录复活"）；不存在则新增。

> `CONFIG_FILE` 可保留作为「最近登录账号」的兼容快照，或彻底废弃（推荐迁移后废弃，见 §7）。

### 4.5 新增 API 蓝图 `backend/account_pool_api.py`

风格对齐 `backend/proxy.py`（蓝图 + jsonify）。注册到 `app.py`：`app.register_blueprint(account_pool_bp)`。

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/account-pool` | 列出所有账号（**脱敏**：不返回完整 cookie/token，只给 `token_preview`、状态、计数、剩余有效期、last_error） |
| GET | `/api/account-pool/summary` | 概要：`{ total, active, cooldown, banned, invalid }`，给侧边栏角标/页面头部用 |
| POST | `/api/account-pool/login` | 启动一次扫码登录（复用 `auth._do_login`，成功后写入池）。即"添加账号"按钮 |
| POST | `/api/account-pool/<id>/relogin` | 对某个失效账号重新扫码登录（登录成功覆盖该 id 的凭证并复活） |
| DELETE | `/api/account-pool/<id>` | 从池中删除该账号 |
| GET | `/api/account-pool/events` | 取走并清空"踢出事件"队列，给前端弹 Toast |

> 登录状态机（scanning/success/failed/qrcode）继续复用 `auth.py` 里的 `_login_state` 与 `/api/auth/login`、`/api/auth/status` 也可；账号池登录可直接复用 `/api/auth/login`，只是成功回调改成写池。实现 Agent 二选一，建议**复用 `auth.py` 的登录流程**，只在成功回调里调 `account_pool.add_or_update`，避免重复 Playwright 代码。

### 4.6 脱敏要求

`GET /api/account-pool` 返回给前端的每个账号**禁止**包含完整 `cookie_str` / `cookies` / 完整 `token`。只返回：

```json
{
  "id": "acc_...",
  "nickname": "某公众号",
  "avatar": "https://...",
  "token_preview": "abcd1234...",
  "status": "active",
  "failures": 0,
  "risk_hits": 0,
  "last_used": 1718000000,
  "cooldown_until": 0,
  "last_error": null,
  "remaining_seconds": 320000,
  "expired": false
}
```

把现有 `LoginPage`（单账号状态卡）改造为 `AccountPoolPage`（多账号网格）。两种落地方式，**推荐方式 A**：

### 5.3 交互逻辑

- **添加账号**：点「+ 添加账号」→ 调 `POST /api/account-pool/login`（复用扫码流程）→ 轮询 `auth.status` 的 `login_state` 显示扫码进度（可弹 Modal 或页内状态区，沿用现有 `startStatusPolling` 逻辑）→ 成功后刷新账号网格。
- **重新登录**（对 banned/invalid 卡片）：调 `POST /api/account-pool/<id>/relogin`，同上扫码流程。
- **删除**：`Modal.confirm` → `DELETE /api/account-pool/<id>` → 刷新。
- **被踢出提示**：账号池页 `init()` 里启动定时器（如每 15s）轮询 `GET /api/account-pool/events`，有事件就 `Toast.warning(...)`。同时建议在全局 `App.checkAuthStatus()`（app.js，每 30s）里也拉一次 events，这样即使用户不在账号池页也能收到提示。

## 6. 风控判定与踢出策略（实现细则）

### 6.1 阈值与冷却（可在 `config.py`/常量集中配置）

| 参数 | 默认 | 说明 |
|------|------|------|
| `COOLDOWN_SECONDS` | 600s（10 分钟） | 单次 `200013` 后冷却时长 |
| `RISK_KICK_THRESHOLD` | 3 | 累计 `200013` 次数达此值 → `banned` |
| `FAILURE_KICK_THRESHOLD` | 8 | 连续普通失败达此值 → `invalid` |
| `LOGIN_VALID_SECONDS` | 4 天 | 凭证过期判断（沿用 auth.py） |

> 这些阈值偏保守可调。建议初版按上表；上线后据实际风控频率微调。

### 6.2 冷却自愈

`acquire()` 每次扫描时，把 `status==cooldown 且 now >= cooldown_until` 的账号自动恢复 `active`（计数不清零，保留 risk_hits 作为"案底"，便于判断是否该升级为 banned）。

### 6.3 过期判断

`acquire()` 跳过 `now - save_time > LOGIN_VALID_SECONDS` 的账号，并将其 `status` 置为 `invalid`、产生一条踢出事件（原因="凭证已过期"）。

### 6.4 并发与锁

- 全局 `self._lock = threading.Lock()`。
- `acquire` / `report` / 增删改全程持锁后再 `_save`（写文件），与代理池 `_proxy_lock` 用法一致。
- 注意：`_save` 写盘在锁内，单机本地应用并发量低，可接受。

## 10. 风险与注意事项

- **风控判定的误伤**：`200013` 也可能是请求过快而非账号本身问题。建议冷却（软踢出）优先，达阈值再硬踢，避免账号被误 ban。阈值 `RISK_KICK_THRESHOLD` 要可调。
- **代理池叠加**：账号池与现有代理池**正交**——一次请求 = 一个账号 × 一个代理。两者的 `report` 互不干扰：HTTP 层失败上报代理池，`base_resp.ret` 层失败上报账号池。实现时不要混淆这两类失败。
- **登录串行**：Playwright 扫码登录是有头浏览器、串行的（`_active_browser` 单实例）。"添加账号"流程要复用现有"同一时刻只允许一个登录中"的互斥（`_login_state.status == "scanning"` 时拒绝）。
- **RSS 后台线程**：`rss_scheduler` 在后台并发调用采集，务必确认 `acquire/report` 的线程安全（已用锁）。
- **不要把账号凭证写进 URL / 日志**：日志只打印 `nickname` 或 `token_preview`，不打印完整 cookie。
